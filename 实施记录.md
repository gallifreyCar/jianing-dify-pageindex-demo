# 实施记录

这份文档记录了本项目在 2026 年 3 月 15 日的主要思路、实施过程、遇到的问题以及解决办法。

## 项目目标

原有系统是老的 RAG 方案，旧知识库已经无法直接导入，也不希望继续依赖原来的向量数据库。

新的目标是：

- 本地部署 Dify
- 使用官方 `VectifyAI/PageIndex` 路线做文档检索
- 用 DeepSeek 作为模型
- 做一套售后客服工作流
- 能区分“文档问题”和“订单查询”
- 文档问题还要继续区分售后场景

## 整体思路

我们最终选择的是一条比较稳的落地路径：

1. 先把 Dify 本地跑起来。
2. 先做一版可体验的客服工作流。
3. 用 mock 文档和 mock 订单接口把“路由逻辑”先跑通。
4. 再把文档检索分支从 mock 接口替换成官方 PageIndex 接入服务。
5. 最后把整个 demo 整理成一个可复现、可提交到 GitHub 的工程。

## 工作流设计

客服工作流采用两层分类：

1. 一级分类
   - 文档问题
   - 订单查询
   - 其他
2. 二级分类
   - 退换货
   - 物流签收
   - 质保售后
   - 安装使用
   - 其他

这样做的原因是：

- 查单问题天然更适合走 API，而不是文档检索
- 售后文档如果不分场景，检索范围会太大，回答容易飘
- 电商客服常见问题高度结构化，分场景后更容易稳定返回客服话术

## 为什么没有继续走旧 RAG

旧方案的核心问题是知识库已经废弃，旧向量数据库无法直接迁移复用。

相比之下，PageIndex 这条路更符合当前需求：

- 可以直接围绕 Markdown/PDF 组织文档
- 不强依赖旧向量索引
- 更适合做结构化文档推理式检索

## 为什么先做 mock

一开始没有真实订单接口，也没有现成可直接导入的新知识库。

如果直接上真实业务系统，会同时遇到这些不确定性：

- Dify 工作流结构是否可导入
- 模型节点是否兼容
- 文档分流是否合理
- PageIndex 接口形态是否适合挂到 Dify
- 订单接口的鉴权和字段定义是什么

所以先做 mock 的意义是：

- 先验证工作流架构是否成立
- 先让你能在本地直接体验
- 后续替换真实接口时只改连接层，不改整体思路

## 遇到的问题与解决过程

### 1. Dify DSL 导入兼容问题

一开始的 DSL 在当前 Dify 版本里存在节点字段兼容问题，导致导入后前端报错或页面异常。

解决方式：

- 多次调整 DSL 版本和 graph 字段结构
- 将分类逻辑收敛到更稳的 `LLM + IF/ELSE` 路线
- 最终导入出一个可正常打开的工作流版本

### 2. Safari 自动化受限

Safari 默认不方便通过脚本直接读取页面错误，导致定位导入问题效率比较低。

解决方式：

- 切换到 Chrome
- 使用浏览器可读页面内容和接口方式确认导入结果

### 3. 官方 PageIndex 仓库不是现成 HTTP 服务

官方开源的 `VectifyAI/PageIndex` 提供的是索引和检索思路，并不是一个开箱即用的本地 HTTP 检索服务。

解决方式：

- 基于官方仓库的 `md_to_tree` 和树检索思路
- 自己封装了一个 `pageindex_service.py`
- 让 Dify 能通过 HTTP 调用官方 PageIndex 路线

### 4. 官方仓库初始目录异常

最开始本地 `PageIndex-official` 目录只有空目录结构，没有完整源码，导致挂载和读取都不正常。

解决方式：

- 重新单独拉取完整官方仓库到 `PageIndex-official-src`
- 后续统一以完整源码目录作为服务挂载源

### 5. PageIndex 默认只适配 OpenAI

官方仓库默认按 OpenAI API 使用，直接接 DeepSeek 会遇到兼容问题。

解决方式：

- 给 `pageindex/utils.py` 补了 OpenAI-compatible `base_url` 支持
- 增加 `tiktoken` 的 fallback，避免 `deepseek-chat` 之类模型名不被识别

### 6. Docker 挂载文件异常

最早把 `requirements.txt` 和源码目录分别挂载到容器里时，Docker 把目标路径识别得不稳定，甚至出现把文件挂成目录的情况。

解决方式：

- 不再做零碎挂载
- 直接挂完整 `PageIndex-official-src` 目录到 `/opt/pageindex`

### 7. 服务启动太慢

一开始 `pageindex_service.py` 在启动时就预构建所有场景树索引，导致健康检查长时间拿不到响应，看起来像服务没起来。

解决方式：

- 改成“按场景懒加载”
- 服务先快速启动
- 第一次请求某个场景时再生成那棵树

### 8. PageIndex 返回结构和外层封装不一致

官方 `md_to_tree` 返回的是包含 `structure` 的对象，而不是单一节点树。外层服务最初少处理了一层结构，导致检索虽然调用成功，但命不中实际文档节点。

解决方式：

- 在服务层增加 `get_tree_root()` 适配
- 检索和节点查找统一对 `structure[0]` 做处理

### 9. 首次检索耗时较长

第一次命中某个场景时，需要同时完成：

- 官方树索引生成
- 检索推理
- 回答生成

所以首查明显比后续请求慢。

解决方式：

- 接受首次冷启动成本
- 通过懒加载把成本延后到真正查询时
- 后续同场景访问速度明显改善

## 最终落地结果

最终交付的是一套本地可体验的 demo：

- Dify 本地可访问
- DeepSeek 已接入
- 售后客服工作流可用
- 订单查询分支可用
- 文档问题分支已切到官方 PageIndex 路线
- mock 文档和 mock 订单号可以直接测试

已验证的例子包括：

- `退款规则是什么？`
- `物流签收破损怎么处理？`
- `帮我查一下订单 JN202603150001`

## 为什么这样整理 GitHub 仓库

最终没有把整套上游 `dify` 和 `PageIndex` 官方源码直接一起提交，而是选择了更轻的方式：

- 保留我们自己的客服工作流和服务代码
- 把对上游仓库的改动整理成 patch

这么做的好处是：

- 仓库更干净
- 结构更清楚
- 便于后续同步上游版本
- 也更适合拿来当项目模板继续改

## 下一步建议

如果后续继续做成真实项目，建议按这个顺序推进：

1. 把 mock 订单接口替换成真实业务 API
2. 把 mock 文档替换成真实售后文档
3. 梳理文档目录结构，统一 Markdown/PDF 格式
4. 为 PageIndex 加缓存和预热
5. 优化客服回答话术
6. 加转人工、收集补充信息、工单回写等能力
